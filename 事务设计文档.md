# 合家云社区物业管理平台
## 事务设计文档

---

## 一、概述

本文档描述合家云社区物业管理平台中核心业务操作的事务设计。事务设计关注特定功能对数据库的一系列操作，保证数据一致性和完整性。

---

## 二、事务设计原则

1. **一致性**：事务执行前后，数据库状态保持一致
2. **原子性**：事务中的操作要么全部成功，要么全部失败回滚
3. **隔离性**：并发事务互不干扰
4. **持久性**：事务提交后，结果永久保存

## 三、核心事务设计

### 3.1 人房绑定事务

**功能描述**：将业主/家属/租客与房屋进行绑定

**事务流程**：

```
START TRANSACTION;

-- 1. 检查是否已存在有效绑定（同一业主、同一房屋）
SELECT COUNT(*) FROM hjy_owner_room
WHERE owner_id = ? AND room_id = ? AND status = 1;

IF count > 0 THEN
    ROLLBACK; -- 已存在绑定，回滚
    RETURN "该业主已绑定此房屋"
END IF;

-- 2. 如果是租客关系，检查该房屋是否已有租客
IF relation_type = 'tenant' THEN
    SELECT COUNT(*) FROM hjy_owner_room
    WHERE room_id = ? AND relation_type = 'tenant' AND status = 1;

    IF count > 0 THEN
        ROLLBACK;
        RETURN "该房屋已有租客，无法重复绑定"
    END IF;
END IF;

-- 3. 插入绑定记录
INSERT INTO hjy_owner_room (owner_id, room_id, relation_type, bind_time, status, create_time)
VALUES (?, ?, ?, NOW(), 1, NOW());

-- 4. 如果是业主关系，更新房屋状态为已入住
IF relation_type = 'owner' THEN
    UPDATE hjy_room SET status = 'occupied', update_time = NOW()
    WHERE id = ?;
END IF;

COMMIT;
```

**关键点**：
- 使用 `@Transactional` 注解保证原子性
- 租客唯一性检查防止一房多租
- 业主绑定时自动更新房屋状态

**异常处理**：任何步骤失败则整体回滚

### 3.2 人房解绑事务

**功能描述**：解除业主/家属/租客与房屋的绑定关系

**事务流程**：

```
START TRANSACTION;

-- 1. 查询当前绑定信息
SELECT room_id, relation_type FROM hjy_owner_room WHERE id = ? AND status = 1;

-- 2. 将绑定状态改为无效
UPDATE hjy_owner_room
SET status = 0, unbind_time = NOW(), update_time = NOW()
WHERE id = ?;

-- 3. 检查该房屋是否还有其他有效绑定
SELECT COUNT(*) FROM hjy_owner_room
WHERE room_id = ? AND status = 1;

IF count == 0 THEN
    -- 没有其他绑定，更新房屋状态为空置
    UPDATE hjy_room SET status = 'vacant', update_time = NOW()
    WHERE id = ?;
END IF;

COMMIT;
```

**关键点**：
- 软删除（修改status）保留历史记录
- 只有当房屋完全没有绑定时，才改为空置状态

### 3.3 删除楼栋事务

**功能描述**：删除楼栋时，需要检查是否有下级单元

**事务流程**：

```
START TRANSACTION;

-- 1. 检查该楼栋下是否有单元
SELECT COUNT(*) FROM hjy_unit WHERE building_id = ?;

IF count > 0 THEN
    ROLLBACK;
    RETURN "该楼栋下存在单元，无法删除"
END IF;

-- 2. 执行删除（ON DELETE RESTRICT 保证上级数据安全）
DELETE FROM hjy_building WHERE id = ?;

COMMIT;
```

**关键点**：
- 数据库外键 `ON DELETE RESTRICT` 兜底保护
- 应用层先检查再删除，用户体验更好

### 3.5 删除房屋事务

**功能描述**：删除房屋时，需检查房屋状态（已入住不允许删除）

**事务流程**：

```
START TRANSACTION;

-- 1. 查询房屋状态
SELECT status FROM hjy_room WHERE id = ?;

IF status == 'occupied' THEN
    ROLLBACK;
    RETURN "该房屋已入住，无法删除"
END IF;

-- 2. 检查是否有有效人房绑定
SELECT COUNT(*) FROM hjy_owner_room
WHERE room_id = ? AND status = 1;

IF count > 0 THEN
    ROLLBACK;
    RETURN "该房屋仍有绑定关系，无法删除"
END IF;

-- 3. 执行删除
DELETE FROM hjy_room WHERE id = ?;

COMMIT;
```

**关键点**：
- 已入住房屋禁止删除，防止数据丢失
- 有效绑定存在时也不允许删除

### 3.7 管理员修改用户事务

**功能描述**：管理员修改用户信息时，保护 admin 账号不被随意修改

**事务流程**：

```
START TRANSACTION;

-- 1. 获取当前操作用户（从请求头 X-Username）
-- 2. 查询要修改的用户
SELECT username FROM sys_user WHERE id = ?;

IF target_user.username == 'admin' AND current_user != 'admin' THEN
    ROLLBACK;
    RETURN 403 "无权限修改 admin 账号"
END IF;

-- 3. 执行更新
UPDATE sys_user SET real_name = ?, phone = ?, email = ?, update_time = NOW()
WHERE id = ?;

COMMIT;
```

**关键点**：
- 通过请求头传递当前用户信息
- admin 账号只能自己修改

## 四、事务隔离级别

本系统采用 MySQL 默认隔离级别 **REPEATABLE READ**，满足以下场景：

| 场景 | 隔离级别支持 |
|------|------------|
| 并发绑定同一房屋的不同业主 | ✅ 第二个绑定会失败（检查重复）|
| 并发解绑和绑定同一房屋 | ✅ 事务串行执行 |
| 并发删除同一楼栋 | ✅ 只有一个成功 |

## 五、并发控制

### 5.1 乐观锁策略
对于人房绑定等高并发场景，采用检查后再插入的方式：

```java
// 检查是否存在 → 存在则拒绝，不阻塞其他操作
if (existCount > 0) {
    return CommonResult.error("该业主已绑定此房屋");
}
```

### 5.2 数据库约束
- 外键约束保证层级数据完整性（`ON DELETE RESTRICT`）
- 唯一索引防止重复绑定
- 状态字段实现软删除

## 六、事务失效场景及处理

| 可能失效的场景 | 处理方式 |
|--------------|---------|
| 方法内部调用（this.xxx()）| 注入自身代理，或拆分为两个方法 |
| 非 public 方法加 @Transactional | 改为 public 方法 |
| 异常被 catch 吞掉 | 重新抛出异常，或使用 TransactionTemplate |
| 异步方法 | 事务无法传播到异步线程，需单独处理 |

本系统使用 `@Transactional` 注解方式，默认只对 RuntimeException 和 Unchecked Exception 回滚。
