# 合家云社区物业管理平台
## 中期检查报告

---

## 一、项目简介

**项目名称**：合家云社区物业管理平台

**项目背景**：为社区物业管理部门设计的管理信息系统，实现对社区资产（小区、楼栋、单元、房屋）和业主信息的统一管理，支持多角色权限控制和数据统计。

**开发形式**：B/S 架构，Spring Boot + Vue.js + MySQL

---

## 二、架构选型

### 2.1 后端技术选型

| 技术 | 版本 | 说明 |
|------|------|------|
| JDK | 21 | 长期支持版本，现代化语法特性 |
| Spring Boot | 3.2.0 | 生态成熟，约定优于配置 |
| MyBatis-Plus | 3.5.5 | 简化 CRUD，自动生成 SQL，内置分页插件 |
| MySQL | 8.0 | 课程指定，兼容性好 |
| Maven | 3.x | 项目构建和依赖管理 |

**选型理由**：
- Spring Boot 提供快速开发体验，内嵌 Tomcat，部署简单
- MyBatis-Plus 相比 MyBatis 减少 80% 的 SQL 编写量，适合课程设计进度要求
- MySQL 8.0 支持窗口函数、CTE 等高级特性

### 2.2 前端技术选型

| 技术 | 版本 | 说明 |
|------|------|------|
| Vue.js | 3.x | 渐进式框架，组件化开发，文档完善 |
| Vite | 5.x | 新一代构建工具，开发启动快，热更新优秀 |
| Element Plus | 2.x | Element UI 的 Vue 3 版本，组件丰富 |
| Axios | 1.x | HTTP 请求库，支持请求/响应拦截器 |

**选型理由**：
- Vue 3 Composition API 使代码组织更清晰，逻辑复用更容易
- Vite 相比 Webpack 大幅提升开发体验
- Element Plus 组件覆盖管理系统的绝大部分场景

## 四、分层架构

### 4.1 后端分层

```
Controller层（控制层）
    ↓ 接收HTTP请求，参数校验
Service层（业务逻辑层）
    ↓ 调用业务逻辑
Mapper层（数据访问层）
    ↓ 操作数据库
MySQL数据库
```

**各层职责**：

| 层级 | 所在包 | 职责 |
|------|--------|------|
| Controller | `controller/` | 接收请求、参数校验、调用Service、返回响应 |
| Service接口 | `service/` | 定义业务方法，隔离实现 |
| Service实现 | `service/impl/` | 编写业务逻辑，事务管理 |
| Mapper | `mapper/` | 继承BaseMapper，调用MyBatis-Plus方法 |
| Entity | `model/entity/` | 对应数据库表结构 |
| CommonResult | `model/common/` | 统一响应结果封装 |

### 5.2 数据库表汇总（共11张）

| 序号 | 表名 | 说明 | 关联关系 |
|------|------|------|---------|
| 1 | hjy_community | 小区表 | 顶层，无上级 |
| 2 | hjy_building | 楼栋表 | community_id → community |
| 3 | hjy_unit | 单元表 | building_id → building |
| 4 | hjy_room | 房屋表 | unit_id → unit |
| 5 | sys_user | 用户表 | 无外键依赖 |
| 6 | sys_role | 角色表 | 无外键依赖 |
| 7 | sys_menu | 菜单表 | parent_id 自关联 |
| 8 | sys_user_role | 用户角色关联 | user_id→user, role_id→role |
| 9 | sys_role_menu | 角色菜单关联 | role_id→role, menu_id→menu |
| 10 | hjy_owner | 业主档案表 | 无外键依赖 |
| 11 | hjy_owner_room | 人房关联表 | owner_id→owner, room_id→room |

### 5.3 核心表结构

**房屋表（hjy_room）**：
```sql
- id           BIGINT      主键
- unit_id      BIGINT      所属单元ID（FK）
- room_no      VARCHAR(50) 房间号
- floor        INT         楼层
- area         DECIMAL     建筑面积
- room_type    VARCHAR(20) 户型
- status       VARCHAR(20) 状态（vacant/occupied/decorated）
- create_time  DATETIME
- update_time  DATETIME
```

**人房关联表（hjy_owner_room）**：
```sql
- id            BIGINT      主键
- owner_id      BIGINT      业主ID（FK）
- room_id       BIGINT      房屋ID（FK）
- relation_type VARCHAR(20) 关系类型（owner/family/tenant）
- status        TINYINT     状态（1-有效，0-已解绑）
- bind_time     DATETIME    绑定时间
- unbind_time   DATETIME    解绑时间
- create_time   DATETIME
- update_time   DATETIME
```

### 5.4 数据约束

- 社区资产层级约束：删除小区前必须先删除其下楼栋，删除楼栋前必须先删除其下单元，删除单元前必须先删除其下房屋（`ON DELETE RESTRICT`）
- 用户角色关联：`ON DELETE CASCADE`，随用户删除而清除
- 人房关联：解绑时不删除记录，而是修改 `status` 和 `unbind_time` 字段，保持历史数据
