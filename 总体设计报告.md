# 合家云社区物业管理平台 - 总体设计报告

---

## 1. 引言

### 1.1 编写目的
本报告旨在全面阐述“合家云社区物业管理平台”的总体设计思路、系统架构、功能模块划分、数据库规划以及接口设计。本文档将作为项目详细设计、编码实现以及后期测试与维护的重要指导依据。

### 1.2 项目背景
随着社区规模的扩大和物业管理的精细化需求日益增长，传统的手工或半自动物业管理模式已无法满足现代社区的高效运营。合家云社区物业管理平台旨在为社区物业管理部门提供一个B/S架构的集中式信息管理系统，实现资产（小区、楼栋、单元、房屋）和业主信息的高效流转，同时通过角色权限控制（RBAC）保障系统安全。

## 2. 系统架构设计

### 2.1 整体架构选型
系统采用经典的 **前后端分离架构**（B/S模式），前端使用Vue 3进行响应式视图开发，后端使用Spring Boot 3提供RESTful API，数据持久层采用MySQL 8.0。

## 3. 系统功能模块设计

根据需求分析，系统总体划分为四大核心模块。

### 3.1 系统管理模块 (System)
- **用户管理**：负责系统账号的创建、停用、信息修改及角色分配。内置超级管理员保护机制，禁止删除/禁用默认的`admin`账户。
- **角色管理**：维护系统的角色列表（如管理员、普通用户），后续可扩展菜单级别的权限绑定。

### 3.2 社区资产管理模块 (Community)
采用树形层级结构管理不动产数据，存在严格的级联依赖关系（小区 -> 楼栋 -> 单元 -> 房屋）。
- **小区管理**：维护小区基础信息（名称、地址等）。
- **楼栋/单元管理**：基于所属小区建立物理空间结构。
- **房屋管理**：最小资产单元，记录房屋状态（空置、已入住、装修中）。在发生关联操作时（如绑定业主），系统自动流转房屋状态。
- *防误删设计*：对所有包含下级节点的层级数据，实施基于外键或业务逻辑的`RESTRICT`拦截，防止出现孤儿数据。

### 3.3 业主与服务管理模块 (Owner)
- **业主档案管理**：维护业主的个人基本信息（姓名、联系方式、证件等）。
- **人房绑定管理**：管理业主与房屋的M:N关系。包含关系类型标识（业主本人、家属、租客）。
- *业务约束*：同一房屋仅允许存在一名“租客”角色；绑定操作会触发房屋状态由“空置”转为“已入住”。

### 3.4 数据统计分析模块 (Dashboard)
- **全局概览**：在首页大屏展示平台关键指标（小区总数、房屋总数、业主总数）。
- **资产分布统计**：利用聚合查询（`GROUP BY`），按小区维度统计楼栋、房屋的数量分布。
- **房屋状态图表**：统计当前各状态（空置/入住/装修）的比例，为物业招租或管理提供决策依据。

## 4. 数据库总体设计

### 4.1 概念模型设计 (E-R)
系统的核心实体关系表现为清晰的层级与多对多关联：

```text
┌───────────────┐     1:N     ┌───────────────┐     1:N     ┌───────────────┐
│ hjy_community │────────────▶│ hjy_building  │────────────▶│   hjy_unit    │
│    (小区表)   │             │   (楼栋表)    │             │   (单元表)    │
└───────────────┘             └───────────────┘             └───────┬───────┘
                                                                    │ 1:N
                                                            ┌───────▼───────┐
                                                            │   hjy_room    │
                                                            │   (房屋表)    │
                                                            └───────┬───────┘
                                                                    │ 1:N
                                                            ┌───────▼───────┐
                                                            │hjy_owner_room │
                                                            │ (人房关联表)  │
                                                            └───────┬───────┘
                                                                    │ N:1
                                                            ┌───────▼───────┐
                                                            │   hjy_owner   │
                                                            │  (业主档案表) │
                                                            └───────────────┘

┌───────────────┐     1:N     ┌───────────────┐     N:1     ┌───────────────┐
│   sys_user    │────────────▶│ sys_user_role │◀────────────│   sys_role    │
│   (用户表)    │             │(用户角色关联) │             │   (角色表)    │
└───────────────┘             └───────────────┘             └───────────────┘
```

### 4.2 核心表结构概览

全系统共设计 9 张核心数据表：

1. **层级资产表 (4张)**：`hjy_community`, `hjy_building`, `hjy_unit`, `hjy_room`。通过上级ID（如`community_id`, `building_id`）建立父子关系。
2. **业主档案表 (1张)**：`hjy_owner`。
3. **关联业务表 (1张)**：`hjy_owner_room`。核心字段包括`owner_id`, `room_id`, `relation_type`（身份类型）。
4. **系统权限表 (3张)**：`sys_user`, `sys_role`, `sys_user_role`。实现标准RBAC模型。

### 4.3 关键索引与约束设计
- **唯一约束 (UNIQUE)**：
  - `sys_user.username`：保证登录名全局唯一。
  - `sys_user_role.(user_id, role_id)`：防止重复分配相同角色。
  - `hjy_owner_room.(owner_id, room_id, relation_type)`：防止重复绑定同一身份。
- **参照完整性 (Foreign Keys)**：
  - 层级表之间配置外键，删除策略设置为 `ON DELETE RESTRICT`，确保在有下级单元或房屋存在时，不能删除上级楼栋或小区。

## 5. 接口与通信规范设计

### 5.1 RESTful API 规范
前后端采用轻量级的JSON格式通过HTTP协议通信。接口设计遵循REST风格：
- 获取资源：`GET /api/module/resource`
- 创建资源：`POST /api/module/resource`
- 更新资源：`PUT /api/module/resource`
- 删除资源：`DELETE /api/module/resource/{id}`

### 5.2 全局统一响应结构
所有API接口必须返回标准的统一结构，以便前端进行全局拦截与状态处理：
```json
{
  "code": 200,          // 业务状态码 (200=成功, 400=参数错误, 401=未授权, 500=服务器错误)
  "message": "操作成功", // 友好的提示信息
  "data": {             // 实际返回的数据体（对象或数组）
    "id": 1,
    "name": "..."
  }
}
```

### 5.3 权限与会话机制
1. **身份认证**：目前采用轻量级Header校验（模拟Token），前端在每次请求头中携带 `X-Username` 与 `X-User-Role`。
2. **后端拦截**：Spring Boot 配置 `HandlerInterceptor` 全局拦截器，拦截未携带认证头的请求，并对敏感写操作（POST/PUT/DELETE）验证是否具有 `admin` 角色。

## 6. 系统安全与异常处理设计

### 6.1 安全防护机制
- **SQL注入防护**：利用MyBatis-Plus的预编译SQL特性（PreparedStatement），避免拼接SQL引发注入漏洞。
- **越权访问防护**：前端通过Vue Router的全局路由守卫（`beforeEach`）拦截无权页面的跳转；后端通过拦截器阻断无权API调用。
- **敏感操作保护**：硬编码限制对系统初始账号（`admin`）的修改与删除行为。

### 6.2 全局异常处理
后端基于 `@RestControllerAdvice` 构建全局异常处理器：
- 捕获 `MethodArgumentNotValidException` 处理表单校验失败，返回400。
- 捕获 `SQLIntegrityConstraintViolationException` 拦截数据库约束冲突（如重复用户名、违规删除），转化为友好的业务提示。
- 捕获 `Exception` 兜底未知异常，记录日志并返回500系统繁忙。前端 Axios 响应拦截器统一对非200状态进行 Message 弹窗提示。
